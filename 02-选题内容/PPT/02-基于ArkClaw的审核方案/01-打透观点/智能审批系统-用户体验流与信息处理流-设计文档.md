## 一、配置文件设计

### 1.1 个人规则配置文件

每个审批人拥有一份飞书电子表格，作为个人审批规则的唯一数据源。

```
配置文件结构：
├── Sheet1: 总览（固定）
│   ├── 列：流程名称 | 规则 JSON | 最后更新时间 | 规则状态
│   └── 每行对应一个流程的完整规则（JSON 格式）
│
├── Sheet2~N: 按流程表单名称命名的 Sheet 页
│   ├── 列：规则分类说明 | 细分场景 | 规则详细描述 | 执行动作 | 规则状态
│   └── 每条规则一行，人工可读、可编辑，规则状态默认为「待验证」
│
└── 维护方式：用户通过飞书电子表格直接编辑，ArkClaw 可读取和更新
```

**Sheet2~N 字段说明**：

| 字段 | 说明 | 可选值/示例 |
|------|------|-----------|
| 规则分类说明 | 规则所属的校验类型 | 表单校验 / 附件校验 / 交叉校验 |
| 细分场景 | 该规则适用的具体业务场景 | 如「水电费报销」「设备采购」「差旅报销」 |
| 规则详细描述 | 自然语言描述规则逻辑 | 「采购金额超过 5 万元且供应商不在合格名录时，需人工复核」 |
| 执行动作 | 规则命中后的处理方式 | 人工复核 / 驳回 |
| 规则状态 | 规则当前是否生效 | 有效 / 失效 / 待验证（默认） |

### 1.2 系统配置文件（config.json）

```json
{
  "user_id": "ou_xxx",

  "rule_spreadsheet": {

    "auth_type": "oauth2"
  },
  "polling": {

    "enabled": true
  }
}
```

```
│  │    ├─ 不存在 → 提示用户"尚未配置个人规则表，是否初始化？"  │      │
│  │    └─ 无权限 → 提示用户"无权限访问规则表，请检查分享设置"   │      │

│                         │                                       │
│                         ▼                                       │

│  ┌──────────────────────────────────────────────────────┐      │
│  │ ① 读取 Sheet1 总览，获取所有已配置流程列表               │      │
│  │ ② 逐流程检查：                                        │      │
│  │    · Sheet 页是否存在且命名匹配                         │      │
│  │    · Sheet 页是否有内容（非空）                         │      │
│  │    · Sheet1 中的 JSON 是否与 Sheet 页内容一致           │      │
│  │    · JSON 格式是否合法                                 │      │
│  │ ③ 输出有效性报告                                      │      │
│  └──────────────────────────────────────────────────────┘      │
│                         │                                       │
│                         ▼                                       │
│  3.1c 校验具体流程规则                                           │
│  ┌──────────────────────────────────────────────────────┐      │
│  │ ① 定位目标流程的 Sheet 页                              │      │
│  │ ② 读取 Sheet 页内所有规则行（含规则状态）               │      │
│  │ ③ 调 BPM API：GET /api/workflow/{流程名}/samples       │      │
│  │    获取该流程的样例数据（含表单字段 + 附件）              │      │
│  │ ④ 逐条样例 × 逐条有效规则 → 匹配引擎                   │      │
│  │    · 表单校验：字段值 vs 条件值                        │      │
│  │    · 附件校验：附件存在性 / 完整性检查                   │      │
│  │    · 交叉校验：多字段组合判断                          │      │
│  │ ⑤ 输出匹配结果矩阵                                    │      │
│  │ ⑥ 基于未命中规则，生成优化建议                          │      │
│  │ ⑦ 基于样例数据特征，生成建议新增规则                     │      │
│  │ ⑧ 用户确认后：                                        │      │
│  │    · 更新对应 Sheet 页（新增/修改规则行）                │      │
│  │    · 重新生成规则 JSON                                 │      │
│  │    · 更新 Sheet1 总览中该流程的 JSON 和时间戳            │      │
│  └──────────────────────────────────────────────────────┘      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 3.1c 详细：规则匹配引擎逻辑

```
输入：
  · 样例数据（JSON）：{ "字段1": "值1", "字段2": "值2", ..., "附件": [...] }
  · 规则列表（Array）：[
      { "分类": "表单校验", "细分场景": "设备采购", "描述": "采购金额超过 5 万元", "动作": "人工复核", "状态": "有效" },
      ...
    ]

处理：
  FOR EACH 规则 IN 规则列表（仅处理状态为「有效」的规则）：
    IF 规则.分类 == "表单校验"：
      从规则详细描述中提取字段名和条件
      在样例数据中定位对应字段值
      执行条件判断（> / < / = / ≠ / 包含 / 不包含 / 为空 / 不为空）
      记录命中/未命中
    IF 规则.分类 == "附件校验"：
      检查样例附件列表中是否存在指定附件
      检查附件 OCR 解析状态（完整/部分/无法解析）
      记录命中/未命中
    IF 规则.分类 == "交叉校验"：
      从规则详细描述中提取多字段组合条件
      解析 AND / OR 逻辑
      逐子条件判断
      记录命中/未命中

输出：
  · 匹配结果矩阵（规则 × 样例）
  · 未命中规则清单 + 原因
  · 优化建议
  · 建议新增规则
```

## 四、异常场景汇总

### 4.1 审批规则管理 — 异常矩阵

| 异常场景 | 触发条件 | 处理策略 | 用户感知 |
|---------|---------|---------|---------|
| 未配置个人规则表 | config.json 中无 rule_spreadsheet | 提示用户初始化 | 「尚未配置个人规则表，是否需要我帮你创建？」 |
| 规则表无权限访问 | 飞书 API 返回 403 | 提示用户检查分享设置 | 「无权限访问规则表，请检查电子表格的分享设置」 |
| Sheet 页不存在 | 查询的流程在 Sheet1 有记录但无对应 Sheet | 提示规则不完整 | 「该流程的规则 Sheet 页不存在，规则可能已损坏」 |
| Sheet 页为空 | Sheet 页存在但无数据行 | 提示规则为空 | 「该流程尚未配置任何规则」 |
| JSON 格式异常 | Sheet1 中 JSON 解析失败 | 提示规则异常，建议重新生成 | 「规则 JSON 格式异常，建议重新校验并更新」 |
| BPM 无样例数据 | BPM API 返回空样例 | 提示无法校验 | 「该流程暂无样例数据，无法进行规则校验」 |
| BPM 接口超时 | API 调用超过 10s | 重试 1 次，仍失败则提示 | 「BPM 接口响应超时，请稍后重试」 |
| 规则写入失败 | 飞书 API 写入返回错误 | 提示具体原因 | 「规则更新失败：{原因}，建议手动编辑」 |
| 用户确认超时 | 校验后 30 分钟未确认 | 结果过期，需重新校验 | 「校验结果已过期，请重新发起校验」 |

### 4.2 单据审核管理 — 异常矩阵

| 异常场景 | 触发条件 | 处理策略 | 用户感知 |
|---------|---------|---------|---------|
| BPM 无权限 | API 返回 401/403 | 提示用户授权 | 「无 BPM 访问权限，请先完成授权」 |
| BPM 接口超时 | API 调用超过 10s | 重试 1 次，仍失败则跳过本条 | 本条不推送，下次轮询重试 |
| 待审核清单为空 | BPM 返回空数组 | 跳过，不推送 | 无感知 |
| 附件下载失败 | 附件 URL 失效或网络异常 | 标记"无法获取"，降级处理 | 卡片标注「附件无法获取」 |
| OCR 解析超时 | OCR 服务超过 30s | 标记"解析超时"，降级处理 | 卡片标注「附件解析超时」 |
| OCR 部分解析 | 置信度 50-80% | 标记"部分解析"，仍参与审核 | 卡片标注「附件部分可读」 |
| OCR 无法解析 | 文件损坏/加密/质量过低 | 标记"无法解析"，跳过该附件 | 卡片标注「附件无法解析」 |
| 不支持附件格式 | .zip/.rar 等非文档格式 | 标记"格式不支持" | 卡片标注「附件格式不支持」 |
| 流程未配置规则 | Sheet1 中无该流程 JSON | 标记"无规则"，跳过审核 | 卡片标注「该流程未配置审批规则」 |
| 规则 JSON 解析失败 | JSON 格式异常 | 标记"规则异常"，跳过审核 | 卡片标注「审批规则异常，请联系管理员」 |
| AI 审核超时 | 大模型调用超过 60s | 标记"AI 审核超时"，降级 | 卡片仅展示表单摘要，无 AI 建议 |
| AI 审核返回异常 | 返回格式不符合预期 | 标记"AI 审核异常"，降级 | 卡片仅展示表单摘要，无 AI 建议 |
| BPM 更新审核建议失败 | PATCH 接口返回错误 | 记录日志，仍推送卡片 | 卡片标注「BPM 同步失败」 |
| 飞书卡片推送失败 | 消息接口返回错误 | 重试 1 次 | 无感知（重试后仍失败则记录日志） |
| 用户操作回调失败 | 审批/驳回接口返回错误 | 卡片提示，不自动重试 | 「操作失败，请重试或跳转 BPM 手动处理」 |
| 重复推送 | 同一任务多次轮询到 | 去重：对比上次清单 | 无感知 |
| 并发处理冲突 | 用户手动在 BPM 已处理 | 调 BPM 接口时发现状态已变更 | 卡片标注「该单据已在 BPM 中处理」 |

## 五、关键设计决策

| 决策点 | 选择 | 理由 |
|-------|------|------|
| 规则存储方式 | 飞书电子表格（Sheet 页可读 + Sheet1 JSON 可机读） | 人机双可读，降低维护门槛 |
| 规则分类 | 表单校验 / 附件校验 / 交叉校验 | 覆盖单字段、文件、多字段组合三类场景 |
| 规则状态管理 | 有效 / 失效 / 待验证（默认） | 新增规则默认待验证，校验通过后变有效，避免误伤 |
| 规则更新流程 | 校验 → 建议 → 用户确认 → 写入 | 规则是审批的核心依据，必须人工确认 |
| 审核处理顺序 | 逐条串行（从前往后） | 避免并发更新 BPM 导致冲突 |
| 附件 OCR 策略 | 尽力解析 + 状态标识 | 不因单个附件失败阻塞整体流程 |
| 定时轮询间隔 | 30 分钟 | 平衡时效性与 BPM 接口压力 |
| 用户操作失败 | 不自动重试 | 审批操作不可逆，避免重复提交 |
| AI 审核降级 | 超时/异常时仅展示表单摘要 | 保证基本可用性，不阻塞人工审核 |
