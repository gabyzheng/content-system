# 阿迪达斯知识引擎投标方案 · 打透观点

> 一期聚焦 EC 团队知识赋能 | 品牌视角未来延展 | 2026.06.16

---

## 一、客户画像与核心矛盾

**客户**：阿迪达斯（Adidas）EC（E-Commerce）团队
**核心矛盾**：EC 团队日常运营依赖大量产品知识（商品主档、USP 卖点、PEP 详情页、检测报告），但这些知识分散在邮件、本地文档、各系统中，缺乏统一的知识底座。导致信息传递靠人、内容生产靠人、质量校验靠人、数据对比靠人——**人效瓶颈成为业务瓶颈**。

**一句话总结**：EC 团队不缺知识，缺的是让知识「找得到、用得对、产得快、比得准」的能力。

## 二、五大痛点 × 知识引擎标准能力映射

### 痛点 1：产品资料分散，缺少统一数据源

**现状**：
- 各 Team 关键产品信息分散在邮件、本地文档、各系统中，系统自动化不够
- 商品检测报告无快速查询窗口，靠单文件存储查看
- 源头信息无法完全对齐，功能词表靠人工逐一排查

**知识引擎解法**：

| 痛点维度 | 知识引擎能力 | 具体方案 |
|---------|------------|---------|
| 信息分散 | **统一知识入口** | 邮件、文档、系统数据统一接入知识引擎，形成 EC 团队单一知识底座 |
| 检测报告难查 | **分类打标 + 统一检索** | 检测报告按商品/SKU/批次分类入库，支持关键词检索和条件筛选 |
| 源头信息不对齐 | **入库质量管理** | 错别字检测、重复检测、截断检测，确保入库知识质量 |
| 功能词表人工排查 | **标签体系 + 知识挖掘** | 自动提取功能词、属性词，建立标准化词表，替代人工逐一排查 |
| 系统自动化不足 | **API 路由 / 统一检索入口** | 知识引擎提供 API 接口，EC 各业务系统可统一调用知识检索能力 |

**核心价值**：从「人找知识」→「系统推送知识」，信息传递从异步人工对齐 → 实时自动同步。

---

### 痛点 2：主档信息检查繁琐

**现状**：
- 商品主档核心信息（Model Name、Division、Brand-OC、Category Update、Description in Chinese、Colour-OC、Colour in Chinese、Local Product Type、Gender Update、Material_CN、材质-鞋面等 10+ 字段）依赖人工手动创建与逐条校对
- 流程标准化程度极低
- 人工审核极易出现信息错漏，错误通过 PID 快速同步至全平台，造成大范围影响

**知识引擎解法**：

| 痛点维度 | 知识引擎能力 | 具体方案 |
|---------|------------|---------|
| 人工逐条校对 | **知识库 + 规则引擎** | 将主档字段规范（格式、取值范围、关联约束）沉淀为知识库规则，系统自动校验 |
| 标准化程度低 | **知识运营 SOP** | 建立主档信息标准化流程：字段定义 → 校验规则 → 质量门禁 → 发布审核 |
| 信息错漏 | **质量检测** | 入库前自动检测错别字、字段缺失、格式不符、跨字段逻辑矛盾 |
| 错误扩散快 | **版本管理 + 生命周期** | 主档变更可追溯，错误可快速回滚；知识有效期管理确保过期信息自动下线 |

**核心价值**：主档校验从「人工逐条 → 系统自动」，错误拦截在入库前而非发布后。10+ 字段的校验规则一次配置、持续生效。

---

### 痛点 3：USP 规则复杂，依赖人工

**现状**：
- 文案自动化率 **0%**
- USP 文案撰写耗时 **2 小时/个**，需大量人力
- 可能引发卖点遗漏、文案描述不对导致客诉风险

**知识引擎解法**：

| 痛点维度 | 知识引擎能力 | 具体方案 |
|---------|------------|---------|
| 自动化率 0% | **知识引擎 + AI 生成** | 将 USP 规则（品类卖点库、禁用词、合规要求）结构化存入知识引擎，AI 基于规则自动生成文案 |
| 耗时 2H/个 | **规则模板 + 批量生成** | 按品类（鞋/服/配件）建立 USP 模板，输入商品主档信息即可批量生成初稿 |
| 卖点遗漏 | **知识库约束** | 将每个品类的必填卖点维度（材质、功能、设计、场景）作为生成约束条件，确保不遗漏 |
| 客诉风险 | **合规知识库 + 内容审核** | 禁用词库、夸大宣传规则、广告法合规要求入库，生成文案自动过审 |

**核心价值**：USP 文案从「0% 自动化 → 规则驱动的 AI 辅助生成」，2 小时/个 → 分钟级。卖点不遗漏、合规有保障。

---

### 痛点 4：PEP 生成效率低

**现状**：
- PEP 自动化率 **不足 10%**
- 制作仍停留在铺人阶段
- 难以应对高频次的集中制作需求

**知识引擎解法**：

| 痛点维度 | 知识引擎能力 | 具体方案 |
|---------|------------|---------|
| 自动化率 <10% | **知识引擎 + AI 内容生成** | PEP 模板、商品主档、USP 文案、视觉素材统一接入知识引擎，AI 自动组装生成详情页 |
| 铺人阶段 | **知识底座驱动自动化** | 商品信息（主档+USP+检测报告+素材）全部结构化存储，PEP 生成从「人找信息再组装」→「系统自动取数组装」 |
| 高频集中制作 | **批量生成 + 模板管理** | 支持按品类/季节/活动批量生成 PEP，模板统一管理、一键更新 |

**核心价值**：PEP 从「人驱动」→「知识驱动」，自动化率从 <10% → 目标 70%+。高频上新场景不再受人力瓶颈制约。

---

### 痛点 5：商品对比依赖人工，质检报告分析耗时

**现状**：
- 完全由人工进行制作，无任何自动化辅助
- 工作流程：人工根据产品属性 → 查阅各产品间的质检报告 → 逐个数据计算功能参数提升是否符合描述
- 质检报告以完整 PDF 格式存储在 adifox 系统
- 每份报告检测方向不同，能提炼、对比的功能点各不相同
- 整个过程十分耗费人力

**知识引擎解法**：

| 痛点维度 | 知识引擎能力 | 具体方案 |
|---------|------------|---------|
| PDF 未结构化 | **知识入库 + 智能解析** | 质检报告 PDF 入库时自动解析，提取功能参数、检测值、标准值为结构化字段 |
| 检测方向不统一 | **分类打标 + 标签体系** | 按检测方向（物理性能/化学成分/安全标准等）对报告分类打标，建立统一的功能参数标签体系 |
| 对比靠人脑 | **规则引擎 + 自动对比** | 将"功能参数提升是否符合描述"的对比逻辑规则化，系统自动计算并标记差异 |
| 数据孤岛 | **统一检索 + API 路由** | 质检报告与商品主档通过 SKU/产品 ID 关联，一键调取该产品所有相关报告和参数 |
| 缺乏知识复用 | **知识沉淀 + 版本管理** | 历史对比结果自动归档，同类产品可直接复用对比模板和结论 |

**核心价值**：从「人读 PDF → 人算参数 → 人做对比」变为「系统解析 → 自动提取 → 规则对比」。不是没有数据，是数据锁在 PDF 里、散在不同检测方向里、靠人一条条抠出来算。

## 四、品牌视角 · 未来场景延展

一期聚焦 EC 团队，但知识引擎的价值不止于 EC。从品牌全链路视角，知识引擎可以逐步覆盖：

### 4.1 从 EC → 品牌全链路

```
         ┌─────────┐
         │  一期    │  EC 团队知识赋能
         │  EC团队  │  主档校验 · USP生成 · PEP生成 · 检测报告检索 · 商品对比
         └────┬────┘
              │
         ┌────▼────┐
         │  二期    │  产品企划 & 供应链
         │  产品链  │  产品规格知识库 · 材质标准库 · 供应商合规知识
         └────┬────┘
              │
         ┌────▼────┐
         │  三期    │  零售 & 营销
         │  零售链  │  门店话术知识 · 活动方案库 · 消费者洞察知识
         └────┬────┘
              │
         ┌────▼────┐
         │  四期    │  品牌全域
         │  品牌链  │  品牌资产库 · 全球合规知识 · 跨区域知识协同
         └─────────┘
```

### 4.2 各阶段延展场景

| 阶段 | 场景 | 知识引擎价值 |
|------|------|------------|
| **一期 · EC** | 商品主档校验、USP 生成、PEP 生成、检测报告检索、商品对比 | 知识底座 + 质量保障 + AI 生成 + 自动对比 |
| **二期 · 产品链** | 产品规格管理、材质标准库、供应商合规审核 | 标准化知识库 + 规则引擎 + 合规校验 |
| **三期 · 零售链** | 门店培训话术、活动方案复用、消费者反馈分析 | 知识沉淀 + 智能推荐 + 经验复用 |
| **四期 · 品牌链** | 品牌资产统一管理、全球合规知识协同、跨区域最佳实践共享 | 全域知识底座 + 多语言 + 跨组织协同 |

### 4.3 品牌视角的核心叙事

> 阿迪达斯作为全球领先的运动品牌，产品知识是品牌的核心资产。从一双鞋的材质、功能、设计故事，到一个季节的营销主题，再到全球市场的合规要求——这些知识今天散落在邮件、文档、人脑中。
>
> 知识引擎的使命不是替代任何一个系统，而是成为品牌的「知识中枢」——让每一份产品知识被结构化地沉淀、高质量地管理、智能化地应用。从 EC 团队起步，逐步覆盖产品链、零售链、品牌链，最终实现品牌全域知识的统一管理与智能驱动。

## 五、关键待论证事项

1. **USP 规则库冷启动**：阿迪达斯各品类的 USP 规则是否已有文档化？规则库构建需要多少人工投入？[待论证]
2. **PEP 自动化率目标**：从 <10% 提升到多少？不同品类（鞋/服/配件）的自动化难度差异？[待论证]
3. **主档字段校验规则**：10+ 字段的校验规则（格式、取值范围、跨字段约束）是否已有明确定义？[待论证]
4. **质检报告解析精度**：adifox 中 PDF 质检报告的格式是否统一？智能解析的准确率目标？[待论证]
5. **与现有系统对接**：EC 团队当前使用的系统清单（NOVA、SHAREPOINT、ASSETS HUB、PID、ADIFOX、PLATFORM）及对接方式？[待论证]
6. **多语言支持**：阿迪达斯 Global 业务涉及多语言（中/英/德等），知识引擎的多语言知识管理能力？[待论证]
