# 阿迪达斯知识引擎投标方案 · 打透观点

> 一期聚焦 EC 团队知识赋能 | 2026.06.16

---

## 一、客户画像与核心矛盾

**客户**：阿迪达斯（Adidas）EC（E-Commerce）团队
**核心矛盾**：EC 团队日常运营依赖大量产品知识（商品主档、USP 卖点、PEP 详情页、检测报告），但这些知识分散在邮件、本地文档、各系统中，缺乏统一的知识底座。导致信息传递靠人、内容生产靠人、质量校验靠人——人效瓶颈成为业务瓶颈。

**一句话总结**：EC 团队不缺知识，缺的是让知识「找得到、用得对、产得快」的能力。

---

## 二、四大痛点 × 知识引擎能力映射

### 痛点 1：产品资料分散，缺少统一数据源

**现状描述**：
- 各 Team 的关键产品信息分散在邮件、本地文档、各系统中
- 商品检测报告无快速查询窗口，靠单文件存储查看
- 源头信息无法完全对齐，功能词表靠人工逐一排查
- 系统自动化不够，信息传递效率低

**知识引擎解法**：

| 痛点维度 | 知识引擎能力 | 具体方案 |
|---------|------------|---------|
| 信息分散 | **统一知识入口** | 将邮件、本地文档、系统数据统一接入知识引擎，形成 EC 团队单一知识底座 |
| 检测报告难查 | **分类打标 + 统一检索** | 检测报告按商品/SKU/批次分类入库，支持关键词检索和条件筛选 |
| 源头信息不对齐 | **入库质量管理** | 错别字检测、重复检测、截断检测，确保入库知识质量 |
| 功能词表人工排查 | **知识挖掘 + 标签体系** | 自动提取功能词、属性词，建立标准化词表，替代人工逐一排查 |
| 系统自动化不足 | **API 路由 / 统一检索入口** | 知识引擎提供 API 接口，EC 各业务系统可统一调用知识检索能力 |

**核心价值**：从「人找知识」变为「系统推送知识」，信息传递从异步人工对齐变为实时自动同步。

---

### 痛点 2：主档信息检查繁琐

**现状描述**：
- 商品主档核心信息（Model Name、Division、Brand-OC、Category Update、Description in Chinese、Colour-OC、Material_CN、材质-鞋面等 10+ 字段）依赖人工手动创建与逐条校对
- 流程标准化程度极低
- 人工审核极易出现信息错漏
- 错误通过 PID 快速同步至全平台，造成大范围影响

**知识引擎解法**：

| 痛点维度 | 知识引擎能力 | 具体方案 |
|---------|------------|---------|
| 人工逐条校对 | **知识库 + 规则引擎** | 将主档字段规范（格式、取值范围、关联约束）沉淀为知识库规则，系统自动校验 |
| 标准化程度低 | **知识运营 SOP** | 建立主档信息标准化流程：字段定义 → 校验规则 → 质量门禁 → 发布审核 |
| 信息错漏 | **质量检测（错别字/重复/截断）** | 入库前自动检测错别字、字段缺失、格式不符、跨字段逻辑矛盾 |
| 错误扩散快 | **版本管理 + 生命周期** | 主档变更可追溯，错误可快速回滚；知识有效期管理确保过期信息自动下线 |

**核心价值**：主档校验从「人工逐条 → 系统自动」，错误拦截在入库前而非发布后。10+ 字段的校验规则一次配置、持续生效。

---

### 痛点 3：USP 规则复杂，依赖人工

**现状描述**：
- 文案自动化率 **0%**
- USP 文案撰写耗时 **2 小时/个**，需大量人力
- 可能引发卖点遗漏、文案描述不对导致客诉风险

**知识引擎解法**：

| 痛点维度 | 知识引擎能力 | 具体方案 |
|---------|------------|---------|
| 自动化率 0% | **知识引擎 + AI 生成** | 将 USP 规则（品类卖点库、禁用词、合规要求）结构化存入知识引擎，AI 基于规则自动生成文案 |
| 耗时 2H/个 | **规则模板 + 批量生成** | 按品类（鞋/服/配件）建立 USP 模板，输入商品主档信息即可批量生成初稿 |
| 卖点遗漏 | **知识库约束** | 将每个品类的必填卖点维度（材质、功能、设计、场景）作为生成约束条件，确保不遗漏 |
| 客诉风险 | **合规知识库 + 内容审核** | 禁用词库、夸大宣传规则、广告法合规要求入库，生成文案自动过审 |

**核心价值**：USP 文案从「0% 自动化 → 规则驱动的 AI 辅助生成」，2 小时/个 → 分钟级。卖点不遗漏、合规有保障。

---

### 痛点 4：PEP 生成效率低

**现状描述**：
- PEP 自动化率 **不足 10%**
- 制作仍停留在铺人阶段
- 难以应对高频次的集中制作需求

**知识引擎解法**：

| 痛点维度 | 知识引擎能力 | 具体方案 |
|---------|------------|---------|
| 自动化率 <10% | **知识引擎 + AI 内容生成** | 将 PEP 模板、商品主档、USP 文案、视觉素材统一接入知识引擎，AI 自动组装生成详情页 |
| 铺人阶段 | **知识底座驱动自动化** | 商品信息（主档+USP+检测报告+素材）全部结构化存储，PEP 生成从「人找信息再组装」变为「系统自动取数组装」 |
| 高频集中制作 | **批量生成 + 模板管理** | 支持按品类/季节/活动批量生成 PEP，模板统一管理、一键更新 |

**核心价值**：PEP 从「人驱动」变为「知识驱动」，自动化率从 <10% → 目标 70%+。高频上新场景不再受人力瓶颈制约。

## 三、知识引擎对 EC 团队的核心价值主张

```
┌─────────────────────────────────────────────────────────┐
│                    知识引擎 · EC 知识底座                  │
├─────────────────────────────────────────────────────────┤
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐ │
│  │ 统一入口  │  │ 质量保障  │  │ 智能生成  │  │ 持续运营  │ │
│  │          │  │          │  │          │  │          │ │
│  │ 邮件     │  │ 错别字   │  │ USP文案  │  │ 版本管理  │ │
│  │ 文档 ──→ │  │ 重复 ──→ │  │ PEP详情 ──→ │ 有效期 ──→ │ │
│  │ 系统     │  │ 截断     │  │ 检测报告  │  │ 生命周期  │ │
│  │ 检测报告  │  │ 字段校验  │  │ 合规审核  │  │ 分类打标  │ │
│  └──────────┘  └──────────┘  └──────────┘  └──────────┘ │
├─────────────────────────────────────────────────────────┤
│  知识入库 → 分类打标 → 质量检测 → 统一检索 → AI 应用      │
│                    知识运营 SOP                          │
└─────────────────────────────────────────────────────────┘
```

**三层价值递进**：

1. **知识归集层**：分散信息 → 统一知识底座（解决痛点 1）
2. **质量保障层**：人工校验 → 自动质检 + 规则引擎（解决痛点 2）
3. **智能应用层**：人工生产 → AI 辅助生成 USP/PEP（解决痛点 3、4）

## 四、差异化优势（vs. 传统方案）

| 维度 | 传统方案（文档管理/SharePoint） | 知识引擎 |
|------|------------------------------|---------|
| 知识组织 | 文件夹层级，靠人分类 | 分类打标 + 标签体系，多维检索 |
| 质量保障 | 无/人工抽查 | 入库自动质检（错别字/重复/截断/字段校验） |
| 知识应用 | 人找文档 → 人读 → 人写 | API 检索 → AI 理解 → AI 生成 |
| 运营管理 | 无生命周期管理 | 有效期配置、版本日志、周期更新提醒 |
| 与现有系统关系 | 替代 SharePoint | **不替代**，做统一知识入口层，通过 Microsoft Search API 中转（短期）/ 深度集成（长期） |

## 五、关键待论证事项

以下事项需要在投标方案中明确或标注【待论证】：

1. **SharePoint 集成深度**：短期通过 Microsoft Search API 中转可实现统一检索，但无法做 chunk 索引和语义检索。长期深度集成的时间表和方案？[待论证]
2. **USP 规则库构建**：阿迪达斯各品类的 USP 规则是否已有文档化？规则库冷启动需要多少人工投入？[待论证]
3. **PEP 自动化率目标**：从 <10% 提升到多少？不同品类（鞋/服/配件）的自动化难度差异？[待论证]
4. **主档字段校验规则**：10+ 字段的校验规则（格式、取值范围、跨字段约束）是否已有明确定义？[待论证]
5. **与现有系统对接**：EC 团队当前使用的系统清单（PID、商品管理系统、检测报告系统等）及对接方式？[待论证]
